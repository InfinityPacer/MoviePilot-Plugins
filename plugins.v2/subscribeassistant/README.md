# 订阅助手

多场景管理订阅，实现订阅种子删除以及自动待定 / 自动暂停 / 自动洗版，并提供识别增强能力，
拦截下载阶段易出错的同名互串资源（真人 ↔ 动漫、电影 ↔ 剧集等）。

> 兼容性：MoviePilot v2.x（部分能力需要 v2.2.1+ / v2.2.7+，详见"配置说明"备注列）
> 适用场景：依赖 TMDB 作为订阅数据源；通知能力依赖 MoviePilot 已配置的通知渠道

## 功能概览

- **种子删除**：监听下载超时、手动删除、Tracker 响应关键字，自动清理订阅产生的下载任务，并可选触发搜索补全。
- **订阅待定**：对剧集订阅按"天数 / 已下载集数"自动进入待定，避免订阅元数据未就绪时下载错乱；可补写剧集默认总集数。
- **订阅暂停**：按上映 / 即将播出 / 长时间无下载等条件自动暂停或完成，并支持按用户白名单暂停新增订阅。
- **订阅洗版**：按 CRON 周期检查洗版条件；支持分集洗版回填已存在集、洗版资源整理后清理旧版本文件。
- **识别增强**：在订阅候选下载入口对资源做启发式二次识别，拦截同名互串（真人 ↔ 动漫、电影 ↔ 剧集等）。

不在范围内：
- 仅支持 **TMDB 数据源**，不覆盖豆瓣 / Bangumi 等其它来源的订阅。
- 不替代 MoviePilot 主程序的订阅识别与下载链路，只在其前后做拦截 / 清理 / 状态修正。

## 定时服务

| 服务名 | 触发方式 | 周期 / CRON | 说明 |
| --- | --- | --- | --- |
| 下载检查 | interval | 由"下载检查周期"决定（默认 5 分钟） | 巡检下载任务，处理下载超时删除、手动删除监听、Tracker 响应监听、下载中待定等场景；仅当对应能力开启时注册 |

## 深入说明

<a id="cfg-recognition_guard_mode"></a>
#### 识别增强模式（`recognition_guard_mode`）

> ⚠ **实验性功能**，可能误过滤合法订阅；先用「观察记录」模式跑一段时间，确认拦截策略稳定后再切到「保守拦截」/「严格拦截」。

- **解决什么问题**：MoviePilot 默认按种子名 + TMDB 命中下载，遇到同名真人 / 动漫或电影 / 剧集互串时容易拉错资源。开启识别增强后，会在订阅候选下载入口做一次启发式二次识别，把疑似互串资源拦下来。
- **可选值**（共 4 档，按拦截强度递增）：
  - `off` 关闭：不做识别增强；
  - `observe` 观察记录：仅记录命中日志，不剔除候选；
  - `conservative` 保守拦截：仅对高置信度风险资源拦截；
  - `strict` 严格拦截：对所有命中规则的资源拦截。
- **什么时候开 / 不开**：订阅来源混杂、动漫剧集较多、踩过同名互串坑的，可以先用「观察记录」跑 1–2 天看日志命中是否符合预期，再视情况切到「保守拦截」，确认稳定后再考虑「严格拦截」。订阅源单一、命中一直稳定的环境保持「关闭」即可，避免误拦截风险。
- **典型联动**：
  - `recognition_guard_tmdb_recheck_mode` 会对疑似资源再走一次 TMDB 识别，能减少误拦截但会增加识别耗时。
  - `recognition_guard_same_name_protection` 默认 `true`，把同名疑似互串纳入保护范围；订阅源稳定的可以关掉。该字段未配置时回落到旧字段 `recognition_guard_same_name_mode`（兼容遗留）。
  - `recognition_guard_target_mode = auto` 按候选自动判定真人 / 动漫；自动判定出错时再考虑改成 `animation` / `live_action` 强制指定。
  - 主程序已能精确识别媒体 ID 的资源不会被启发式规则拦截，无需为此特意关闭识别增强。
- **示例**（追新番常用组合，只列与默认值不同的项）：

  ```text
  recognition_guard_mode = conservative     # 默认 off
  ```
- **常见误用**：
  - 一上来直接选「严格拦截」并把 TMDB 二次识别开到「观察 + 保守 + 严格」：没经过观察期，命中和误拦截都看不出来，且识别链路会明显变慢。
  - 订阅源单一的环境强行选「严格拦截」：命中率几乎为 0，只增加日志噪音。

<a id="cfg-auto_best_backfill_priority"></a>
#### 回填已存在集（`auto_best_backfill_priority` / `backfill_best_version_now`）

- **解决什么问题**：开启分集洗版后，媒体库里有些集其实已经是"顶档"，但插件并不知道——下一轮洗版巡检还会去下载同档资源，造成重复下载。回填会把"媒体库已存在集"统一标为 `priority=100`（顶档），洗版巡检看到顶档就不再下载。
- **什么时候开 / 不开**：媒体库里已经攒了大量已洗集、又打算继续开分集洗版的，先勾一次「立即扫描存量并回填」(`backfill_best_version_now`)，再常开「回填已存在集」(`auto_best_backfill_priority`) 兜底未来新增订阅。全新订阅 / 没有历史分集洗版数据的，不需要勾「立即扫描存量并回填」，只开「回填已存在集」即可。
- **典型联动**：
  - 仅对启用了分集洗版的订阅生效；纯全集洗版订阅不受影响。
  - 可与「分集转全集」(`auto_best_episode_to_full`) 叠加：先把存量回填为顶档，再走分集到全集切换的流程，不会冲突。
- **示例**（迁库 / 首次启用分集洗版的典型流程）：
  1. 启用插件，配置好分集洗版规则。
  2. 勾选「立即扫描存量并回填」并保存，执行完成会自动取消勾选。
  3. 保留「回填已存在集」常开，处理后续新增订阅。
- **常见误用**：
  - 全新订阅勾「立即扫描存量并回填」：没有存量数据，等于一次空跑。
  - 把「立即扫描存量并回填」当成开关常开：该字段执行完会自动复位，反复勾选只是重复回填同一批数据。

<a id="cfg-download_timeout_progress_threshold"></a>
#### 下载超时进度阈值（`download_timeout_progress_threshold`）

- **解决什么问题**：单纯按"下载多久"判断超时会误杀大体积慢种。当前用的是"进度感知"判定：在「下载超时时间」窗口内，下载**进度增量**低于该阈值（默认 5%）才算停滞；连续多次停滞才会真正删除。
- **什么时候调 / 怎么调**：家宽带宽充裕、刷的是普通体积种子的，保持默认 5%。带宽窄 / 大量大体积种子（蓝光原盘、4K 季合集）的环境把阈值调小（1–2%），否则慢种会被误判停滞。反过来追求高周转的刷流场景可以把阈值调大（10% 左右），让"挪不动"的种子更早被淘汰。
- **典型联动**：
  - 与「下载超时时间」(`download_timeout`) 是一对：窗口越长容错越大；不要单独把窗口拉到很长又把阈值拉到 0。
  - 与「下载连续超时重试次数」(`download_timeout_retry_limit`) 形成"停滞 → 重试 N 次 → 真删除"的串行链路，`retry_limit ≥ 2` 比较稳。
- **示例**（大体积种子友好组合）：

  ```text
  download_timeout = 6                        # 6 小时窗口
  download_timeout_progress_threshold = 2     # 6 小时内涨不到 2% 才算停滞
  download_timeout_retry_limit = 3            # 连续 3 次停滞才真正删除
  ```
- **常见误用**：
  - 阈值设为 0 + 窗口设很短：等价于"过了窗口就删"，大种全部被误杀。
  - 阈值设很高（例如 50%）：几乎没有种子能在窗口内追上进度，全部被判停滞。

<a id="cfg-auto_best_episode_to_full"></a>
#### 分集转全集（`auto_best_episode_to_full`）

- **解决什么问题**：分集洗版可以更精细地命中每一集顶档，但订阅一旦达到目标集数，分集洗版就没必要继续，反而拉长洗版窗口。开启后，订阅目标集数满足时，会从"分集洗版"切换成"全集洗版"，并且让分集洗版在满足后**再进入洗版时限**——而不是分集刚开始就开始倒计时。
- **什么时候开 / 不开**：长期追剧、走分集订阅、希望最终拿到一个高质量合集时开启；只跑一次性洗版（单部电影 / 一次性整季）的不需要开，纯全集洗版即可。
- **典型联动**：
  - 与「洗版时限」(`auto_best_remaining_days`) 配合最有意义：开启此项后，时限从"满足目标集数那一刻"开始计算，而不是从订阅创建时。
  - 与「回填已存在集」可叠加：回填顶档 → 集满切换全集 → 全集时限内完成洗版。
- **示例**（长期追剧分集洗版典型组合）：

  ```text
  auto_best_type = tv_episode
  auto_best_episode_to_full = true
  auto_best_remaining_days = 30
  auto_best_backfill_priority = true
  ```
- **常见误用**：
  - 开了此项但没设「洗版时限」：切换到全集洗版后会一直跑下去，没有自动终止。
  - 在纯电影 / 全集订阅上开：没有"分集到全集"的概念，配置不会有任何效果。

## 使用步骤

1. 在插件市场安装并启用"订阅助手"。
2. 配置 MoviePilot 通知渠道（如需通知），并确保订阅链路使用的是 TMDB 数据源。
3. 按需开启"种子删除 / 订阅待定 / 订阅暂停 / 订阅洗版 / 识别增强"中的一个或多个 Tab。
4. 调整"下载检查周期"和"元数据检查周期"到合适值；洗版用户额外配置"洗版检查周期"CRON。
5. 勾选"立即运行一次"触发一次全量巡检，观察日志确认行为符合预期；如有遗留脏数据，可勾选"重置数据"。

## 注意事项 / 已知风险

- **数据风险**：插件会按规则修改订阅状态、删除种子并可能删除源文件 / 媒体库文件（洗版清理），属于不可逆操作；首次启用前建议备份订阅数据与媒体库索引。
- **仅 TMDB 数据源**：非 TMDB 数据源的订阅可能行为异常，请勿启用。
- **PostgreSQL 用户**：请使用 v2.7.4+，已修复布尔字段写入兼容问题。
- **MoviePilot 版本依赖**：洗版资源整理清理需要 v2.2.7+；`VCronField` 形式的洗版周期需要 v2.2.1+。
- **识别增强（实验性）**：可能误过滤合法订阅；先用「观察记录」模式跑一段时间，确认拦截策略稳定后再切到「保守拦截」/「严格拦截」。
- **洗版清理（实验性）**：`auto_best_clear_history_type` 会清理整理记录并删除源文件 / 媒体库文件，启用前请备份。
- **一次性动作**：`reset_task` / `onlyonce` / `backfill_best_version_now` 执行完会自动复位，无需手工取消勾选。

## 故障排查

- 主日志：`MoviePilot/config/logs/moviepilot.log`
- 插件日志：`MoviePilot/config/logs/plugins/subscribeassistant/`
- 关键过滤词：`订阅助手`、`subscribeassistant`、`download_check`、`meta_check`、`best_version_check`、`recognition_guard`
- 常见问题：
  - 服务列表出现重复或来源错误 → 通过 `GET /api/v1/dashboard/schedule2` 检查 `id`/`provider`/`status`，确认是否因本地插件副本不同步导致重复注册。
  - 设置在 WebUI 改了不生效 → 检查是否被环境变量或旧配置字段覆盖，必要时清掉旧配置后重启插件。
  - 识别增强误拦截 → 切回 "观察" 模式查日志中 `recognition_guard` 的命中原因，再调整关键字 / 年份模式 / 同名保护。
  - 洗版反复下载同档资源 → 启用「回填已存在集」，或勾选一次「立即扫描存量并回填」。

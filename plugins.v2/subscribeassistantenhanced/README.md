# 订阅助手（增强版）

> **BETA 版本提示**：本插件仍处于测试阶段，可能调整订阅状态、洗版记录、下载任务和媒体文件。请先在可回滚环境中验证，再用于长期运行。

多场景管理订阅，实现订阅全生命周期管理。插件覆盖订阅待定、暂停、洗版、删种、远程切换订阅状态、完结守卫和自动纠错等能力。

识别增强不在增强版中提供；需要同名真人 / 动漫、电影 / 剧集互串拦截的环境，请继续使用旧版「订阅助手」的识别增强能力。

> 兼容 MoviePilot v2.13.0 及以上版本
> 适用于使用 TMDB 作为订阅数据源的环境，通知能力依赖 MoviePilot 已配置的通知渠道

- v0.1.9：完结守卫支持关闭、严格、平衡和宽松模式，默认使用平衡策略，并新增本地目标集覆盖（L）信号；当天播出的下一集不再延迟完结，订阅 ID 复用不会继承旧任务状态。所有订阅完成时都会保存纠错快照，默认保留 180 天且按用户配置自动清理；自动恢复按季和剧集组精确匹配，全集洗版文件清理事务超过 72 小时后自动失效。
- v0.1.8：目标范围末集为未来 finale 时不再提前判定完结，仍在播出的剧集会继续按下一集播出间隔暂停。
- v0.1.7：全集洗版清理按 Sxx 季号匹配旧整理记录；删除关联旧种后先等待 5 秒并跨下载器确认 hash，查询失败最多等待 1 分钟、旧种仍存在最多等待 3 分钟，达到上限后继续下载。
- v0.1.6：低置信完结先进入完成前观察，观察期内增集或信号变化会重新判定；观察到期后通过完成快照接管后续增集纠错。分集洗版目标集入库后可在整理完成事件中立即切换为全集洗版，并按订阅目标总集数确认媒体库覆盖。
- v0.1.5：元数据巡检兼容 TMDB 集信息字典形态；新增历史季订阅不再因最后已播日期直接暂停；剧集订阅完成后自动创建全集洗版订阅，确保洗版清理按整季范围执行。
- v0.1.4：统一多来源待定（P）管理。下载发起后保持待定，整理完成后只解除下载造成的待定，其他待定原因继续保留；分集洗版仅在存在多条关联分集下载历史时自动创建洗版订阅。
- v0.1.3：优化订阅集数刷新和定时任务注册日志，新增订阅创建阶段可显示媒体名称、季号、TMDB ID 与触发场景。
- v0.1.2：补齐订阅状态、洗版订阅创建、分集转全集、删种善后与自动纠错重建的通知及事件联动。
- v0.1.0：多场景管理订阅，实现订阅全生命周期管理。

## 功能概览

- **完结守卫**：在主程序准备完成订阅前，按固定信号顺序和所选模式复核当前目标集范围。默认平衡模式会立即接受三集及以上的低置信完结或本地目标集覆盖，短样本继续进入待定观察。
- **订阅待定（P）**：P 是 MoviePilot 的待定状态。开播时间过近、已播集数过少、总集数仍在变化或存在下载任务时，订阅会进入待定（P）。待定期间只搜索已播出的集，所有待定原因解除后恢复启用（R）。
- **订阅暂停**：支持按新增用户、上映时间、播出间隔和长期无下载自动暂停。下载超时删种不会暂停订阅，插件会记录近期删除资源并按配置补搜。
- **下载管理**：跟踪资源下载、下载器任务、手动删种、Tracker 响应和下载进度停滞。删种后会恢复本次下载影响的洗版优先级、记录近期删除资源并按配置补搜。
- **订阅洗版**：按配置自动创建和推进洗版订阅，支持分集转全集、洗版时限、并行下载隔离、已有集回填和旧版本文件清理。
- **自动纠错**：所有订阅完成时都会保存总集数和订阅配置；开启自动纠错后周期性复查 TMDB，发现新增集数时按季和剧集组精确重建订阅并通知。

## 不在范围内
- 仅支持 **TMDB 数据源**，不覆盖豆瓣 / Bangumi 等其它来源的订阅。
- 不提供旧版「识别增强」互串拦截能力。
- 不替代 MoviePilot 主程序订阅识别与下载链路，只在订阅生命周期关键事件前后执行守卫、状态修正、清理和补偿。

## 定时服务

| 服务名 | 触发方式 | 周期 / CRON | 说明 |
| --- | --- | --- | --- |
| 立即运行一次 | date（一次性） | 保存后延迟约 3 秒 | 串行执行元数据检查、下载任务检查、洗版订阅检查和通用巡检；启用「自动纠错」时同时执行自动纠错；仅勾选「立即运行一次」时注册 |
| 元数据检查 | interval | 由「元数据检查周期（小时）」决定，默认 3 小时 | 复核活动订阅元数据，处理上映前 / 播出间隔暂停的双向恢复，以及待定进入 / 退出 |
| 下载任务检查 | interval | 由「下载检查周期（分钟）」决定，默认 10 分钟 | 读取下载器实时状态，处理进度停滞、Tracker 关键字和用户手动删种；仅启用「下载超时自动删除」时注册 |
| 洗版订阅检查 | cron | 由「洗版检查周期」决定，默认 `0 15 * * *` | 推进洗版订阅的完成判定、分集转全集和洗版时限；仅「洗版类型」不是关闭时注册 |
| 自动纠错 | interval | 由「自动纠错间隔（小时）」决定，默认 12 小时 | 复查完成快照中的总集数，发现 TMDB 增集后自动重建订阅；仅启用「自动纠错」时注册 |
| 通用巡检 | interval | 由「通用巡检周期（分钟）」决定，默认 60 分钟 | 依次执行待定超时释放、无下载处理、删除记录清理、完成快照清理和洗版清理事务清理；各子任务互不阻断 |

## 深入说明

<a id="cfg-completion_guard_mode"></a>
#### 完结守卫模式（`completion_guard_mode`）

- **解决什么问题**：主程序认为订阅满足完成条件时，增强版会先检查当前季或剧集组的真实播出范围，避免总集数临时变动、绝对季断档、`mid_season` 标记或下载尚未入库时过早完成。
- **模式选择**：
  - `strict`（严格）：高、中置信立即完成；低置信 I/L 进入完成前观察。
  - `balanced`（平衡，默认）：高、中置信立即完成；目标范围不少于 3 集且不属于高风险范围时，低置信 I/L 立即完成；1 至 2 集或高风险范围继续观察。
  - `loose`（宽松）：低置信 I/L 也立即完成，但季中、总集数不稳定和明确未来集仍会否决。
  - `off`（关闭）：不注册完成守卫，由主程序决定订阅完成。
- **模式与信号裁决**：

  | 信号 / 场景 | 关闭 `off` | 严格 `strict` | 平衡 `balanced` | 宽松 `loose` |
  | --- | --- | --- | --- | --- |
  | 下载任务仍在进行 | 由主程序决定 | 拦截完成，等待整理入库 | 拦截完成，等待整理入库 | 拦截完成，等待整理入库 |
  | M：`mid_season` 季中阶段 | 由主程序决定 | 拦截完成，进入待定（P） | 拦截完成，进入待定（P） | 拦截完成，进入待定（P） |
  | F：总集数近期仍在变化 | 由主程序决定 | 拦截完成，进入待定（P） | 拦截完成，进入待定（P） | 拦截完成，进入待定（P） |
  | 明确未来集：同季下一集播出日晚于当天 | 由主程序决定 | 拦截完成，进入待定（P） | 拦截完成，进入待定（P） | 拦截完成，进入待定（P） |
  | E：剧集已结束 / 已取消 / 目标末集为 finale | 由主程序决定 | 放行完成 | 放行完成 | 放行完成 |
  | I：中置信间接完结（如后续季已出现） | 由主程序决定 | 放行完成 | 放行完成 | 放行完成 |
  | I：低置信间接完结，目标范围不少于 3 集且非高风险 | 由主程序决定 | 进入完成前观察 | 放行完成 | 放行完成 |
  | I：低置信间接完结，1 至 2 集或高风险范围 | 由主程序决定 | 进入完成前观察 | 进入完成前观察 | 放行完成 |
  | L：本地媒体库覆盖目标范围，不少于 3 集且非高风险 | 由主程序决定 | 进入完成前观察 | 放行完成 | 放行完成 |
  | L：本地媒体库覆盖目标范围，1 至 2 集或高风险范围 | 由主程序决定 | 进入完成前观察 | 进入完成前观察 | 放行完成 |
  | 无完结信号且媒体库仍缺目标集 | 由主程序决定 | 拦截完成，进入待定（P） | 拦截完成，进入待定（P） | 拦截完成，进入待定（P） |

  “进入完成前观察”会写入待定（P）并记录观察窗口；窗口到期且信号未变差时才允许后续完成检查放行。所有真正完成的订阅都会通过 `SubscribeComplete` 保存快照，模式只影响完成前是否放行。
- **典型联动**：与「变更速率信号」「播出节奏信号」「季冷却期（天）」「待定超时释放」联动。守卫否决会写入待定（P）并启动释放计时。
- **信号含义**：
  - `SeasonScope` 表示当前订阅季或剧集组对应的统一目标集范围。信号引擎始终基于该范围判断，不直接用主季信息猜测绝对季范围。
  - **季中阶段否决（M）**：最后已播集标记为 `mid_season` 时，说明当前目标范围仍处于阶段中场，直接否决完结。
  - **总集数稳定性（F）**：检查总集数在变更速率窗口内是否稳定。近期仍在变化时否决完结。
  - **明确完结信号（E）**：剧集状态为已结束或已取消，或者目标范围末集带有 `finale` 标记时，确认完结。
  - **季级间接信号（I）**：通过后续季、最后播出集、全量播出和季冷却期等信息判断。绝对季等高风险目标范围进入更保守的分支，不会仅凭低置信冷却期放行。
  - **订阅目标覆盖信号（L）**：订阅 `note` 已覆盖当前目标集范围时生成，不要求 TMDB 标记 `finale`。已下载后从媒体库删除的集不会重新视为未下载，后续增集由完成快照负责发现。
  - **播出节奏辅助（G）**：根据已播集日期估算下一次更新窗口，用于辅助待定释放和超时加速，不单独确认完结。
  - TMDB 的 `next_episode_to_air` 只有播出日期晚于当天时才算未来集；日期等于当天按已经进入播出日处理，日期缺失时保守阻断。
  - 元数据信号链简写为 `M → F → E → I(±high_risk) → G`，完成守卫随后结合 L 和模式裁决。
  - 变更速率只观察 `total_episode` 是否在「变更速率窗口（天）」内发生变化。窗口越长越保守；它不检查每集播出日期，也不能替代开播窗口和低集数保护。
  - 播出节奏根据已播集日期的典型间隔估算下一次更新窗口，等待时间取 `典型间隔 × 节奏窗口系数` 与「节奏窗口下限（天）」中的较大值。播出节奏本身不会直接判定完结，只用于辅助待定释放。
  - 「节奏参与最少集数」要求至少有相应数量的已播集记录才计算节奏；样本不足或播出日期无效时，不使用节奏推算。
  - 「季冷却期（天）」只参与普通目标范围的低置信季级间接信号（I）判断。绝对季、中间出现 `mid_season` 或 `finale` 等高风险范围不会仅凭冷却期放行。
  - 「自动纠错」「自动纠错间隔（小时）」「快照保留（天）」处理完成后的增集。所有订阅完成时都会保存包含总集数的快照；自动纠错开关只控制是否定期复查和重建。
  - 「待定超时释放」「待定超时释放（天）」「按节奏加速释放」只处理完结守卫写入的待定（P）。信号仍不稳定时，待定计时会重新开始；严格模式和未获平衡模式直接放行的低置信完结使用相同窗口完成观察。
  - 低置信观察期间出现 TMDB 增集或完结信号变化时，当前 `guard_veto` 会释放但不会获得完成放行，后续完成检查会按新的数据重新进入观察或恢复搜索。
  - `cadence_expired` 只表示播出节奏窗口已到期，不代表已完结；开启「按节奏加速释放」时，它只会把当前待定释放期限缩短为原来的一半。
- **例子**：一部 12 集连载剧播到第 10 集时，TMDB 短暂把总集数写成 10，主程序可能认为已完成。开启完结守卫后，总集数稳定性（F）和播出节奏辅助（G）仍显示不稳定，订阅会进入待定（P）。后续 TMDB 回到 12 集并继续播出后，待定释放再恢复搜索。
- **常见误用**：关闭完结守卫后，待定（P）和自动纠错仍可运行，但无法拦住主程序的过早完成。

<a id="cfg-pending_download_enabled"></a>
#### 自动待定下载中订阅（`pending_download_enabled`）

- **解决什么问题**：订阅已经选中资源但还没整理入库时，主程序可能先触发完成检查。开启后，增强版会在资源下载事件（`ResourceDownload`）到下载器任务建立事件（`DownloadAdded`）之间保持下载待定。完成守卫看到仍有下载任务时会否决完成，资源选择事件（`ResourceSelection`）也会避免同一集并行下载多个洗版版本。
- **什么时候开或关闭**：普通追剧和洗版建议保持开启。若下载器任务与订阅完成完全由外部流程管理，不希望插件用下载中状态影响完成判断，可以关闭。
- **典型联动**：下载待定会把订阅置为待定（P）。整理完成、下载超时或手动删种会解除对应的下载待定。若剧集信息待确认或完成守卫否决仍然有效，订阅继续保持待定（P）。所有待定原因解除后才恢复启用（R）。
- **内部状态**：下载待定、剧集信息待确认和完成守卫否决分别记录为 `download_pending`、`pending_judge` 和 `guard_veto`，用于避免不同原因互相误释放。
- **常见误用**：下载待定不是长期暂停。未建立下载器任务的待定只覆盖资源下载到下载器任务建立之间的短窗口，超过宽限仍未建立任务时会自动释放。

<a id="cfg-pending_enhanced_enabled"></a>
#### 自动待定剧集订阅（`pending_enhanced_enabled`）

- **解决什么问题**：剧集刚开播、集数很少、TMDB 没有播出日期或总集数仍在变化时，订阅容易按不完整范围搜索并提前完成。待定会让订阅暂时进入待定（P），并在刷新集数时只放出已播出范围。
- **什么时候开或关闭**：追连载剧集、新番、绝对季时建议保持开启。只订阅已完结剧或不希望插件修改订阅状态时再关闭。
- **典型联动**：「剧集待定天数」「剧集待定集数」「待定参考变更速率」决定进入待定的具体条件，「通用巡检周期（分钟）」决定释放检查频率。变更速率只能识别总集数波动，不能替代开播窗口和低集数保护。
- **字段分工**：「剧集待定天数」处理刚开播窗口，「剧集待定集数」处理只播出 1 集等低集数阶段。TMDB 未给总集数时，插件会直接把待定搜索范围锁定为已播集数，不需要用户配置虚拟总集数。
- **常见误用**：待定不会加速完结，只会在信号不稳定时限制搜索范围。总集数缺失时也只开放已播出的集数。

<a id="cfg-pause_enhanced_enabled"></a>
#### 自动暂停订阅（`pause_enhanced_enabled`）

- **解决什么问题**：对暂时不该继续搜索的订阅写入暂停状态，减少无意义搜索。上映前和播出间隔暂停会在条件解除后自动恢复，无下载和用户名暂停属于标记暂停，不会被元数据巡检自动恢复。
- **什么时候开或关闭**：需要自动管理连载空窗、上映前订阅和下载异常的环境建议保持开启。习惯手动管理订阅状态时可以关闭。
- **典型联动**：「即将播出暂停天数」「剧集上映暂停天数」「电影上映暂停天数」控制可自动恢复的暂停，「无下载处理策略」控制长期无下载时的处理方式。
- **常见误用**：下载超时删种不会暂停订阅。近期删除过的同一资源会被跳过，内部通过删除指纹匹配，后续是否补搜由「删除后触发搜索补全」决定。

<a id="cfg-no_download_actions"></a>
#### 无下载处理策略（`no_download_actions`）

- **解决什么问题**：订阅在上映或开播后长时间没有任何下载记录时，可以自动暂停、完成或删除，避免无效订阅长期占用搜索。
- **什么时候开或关闭**：确认订阅源稳定、缺下载通常代表资源不可用时再开启。刚迁移数据或下载历史不完整时先保持默认空策略。
- **动作选择**：同一媒体类型按表单保存顺序取第一个匹配动作。内部动作值包括 `pause_movie`、`pause_tv`、`complete_movie`、`complete_tv`、`delete_movie` 和 `delete_tv`。
- **典型联动**：必须同时配置「电影无下载处理天数」或「剧集无下载处理天数」，天数为 0 时不处理。
- **常见误用**：只选择策略但两个天数字段都保持 0，不会触发任何动作。

<a id="cfg-download_progress_threshold"></a>
#### 下载超时进度阈值（`download_progress_threshold`）

- **解决什么问题**：下载超时不是按总耗时直接删除，而是在「下载超时时间（分钟）」窗口内观察进度增量。低于阈值才算停滞，连续多次停滞后才会删种。
- **什么时候调整**：普通订阅保持默认 10%。大体积 4K、蓝光原盘或慢种较多时可降到 1 至 2，刷流或希望快速淘汰坏种时可适当提高。
- **典型联动**：「下载连续超时重试次数」控制删除前观察多少轮。删种后会记录近期删除资源、解除下载待定，并恢复本次下载影响的洗版优先级。内部使用删除指纹和下载前优先级基线保存这些信息。是否补搜由「删除后触发搜索补全」决定，订阅不会因此暂停。
- **常见误用**：阈值过高会把慢种当作停滞，阈值过低配合很短窗口会让坏种长期占位。

<a id="cfg-auto_check_interval_minutes"></a>
#### 通用巡检周期（分钟）（`auto_check_interval_minutes`）

- **解决什么问题**：待定释放、无下载处理和删除记录清理统一由一个服务按此周期执行。
- **什么时候调整**：默认 60 分钟适合大多数环境。希望状态更快变化时可设为 30 分钟，订阅量较大时可设为 120 或 240 分钟。
- **典型联动**：每轮依次执行已启用的子任务，单个子任务失败不会阻断其他任务。
- **常见误用**：它不控制下载任务、元数据和洗版检查周期。

<a id="cfg-best_version_type"></a>
#### 洗版类型（`best_version_type`）

- **解决什么问题**：控制普通订阅完成后哪些媒体类型可以自动进入洗版流程，同时作为自动洗版总入口。日常使用时通过这个字段选择自动洗版范围。
- **可选值**
  - `no` 表示关闭，不自动创建新的洗版订阅，也不注册洗版完成检查任务。
  - `all` 表示电影和剧集都可自动洗版。
  - `movie` 表示仅处理电影。
  - `tv` 表示仅处理剧集整季。
  - `tv_episode` 表示仅处理分集下载的剧集。只有关联到多条分集下载历史时才自动洗版，单次合集或全集包下载完成不会自动创建洗版订阅。
- **已有洗版订阅**：关闭后不会执行洗版完成检查。如需继续推进已有洗版订阅，需要选择对应媒体范围。
- **典型联动**：「洗版类型」不是 `no` 时，普通订阅完成后会按范围自动创建洗版订阅。当前订阅本身已经是洗版订阅时不会重复创建。
- **常见误用**：把「洗版类型」保持为关闭时，既不会新增洗版订阅，也不会注册洗版完成检查任务。

<a id="cfg-best_version_backfill_enabled"></a>
#### 回填已存在集（`best_version_backfill_enabled` / `backfill_best_version_now`）

- **解决什么问题**：媒体库里已有的集会被标为顶档优先级，后续洗版巡检不再重复下载这些集。普通订阅转洗版、新增洗版订阅和手工扫描存量都可触发回填。
- **什么时候开或关闭**：已有媒体库内容较多、准备启用分集洗版时建议开启。默认关闭，避免首次启用时批量改写存量订阅的按集优先级。
- **典型联动**：「立即扫描存量并回填」只处理已经存在的洗版订阅，执行完成会自动复位。「分集转全集」可在回填后继续判断是否已全量在库。
- **常见误用**：不要把「立即扫描存量并回填」当成长期开关。它是一次性动作，重复勾选只会重复扫描同一批订阅。

<a id="cfg-best_version_episode_to_full"></a>
#### 分集转全集（`best_version_episode_to_full`）

- **解决什么问题**：分集洗版在目标集全部入库后，可以原地切换为全集洗版，后续用整季包继续完成最终洗版。
- **什么时候开或关闭**：长期追剧并希望最终获得整季高质量版本时开启。电影或纯整季洗版不需要开启。
- **典型联动**：「洗版类型」启用时才会推进完成判定。整理完成事件会先尝试即时切换；若未命中或错过事件，洗版巡检仍按「洗版检查周期」兜底推进。「洗版时限（天）」从洗版活动时间计算，分集转全集后仍会参与超时终止。「回填已存在集」能减少重复下载。
- **常见误用**：开启分集转全集但没有任何目标集入库时不会立即切换。即时切换前提是当前订阅仍为分集洗版、主程序缺集字段已归零（`lack_episode=0`），并且媒体库缺集确认结果为空；确认范围按订阅目标总集数计算。

<a id="cfg-best_version_clear_history_type"></a>
#### 清理整理记录范围（`best_version_clear_history_type`）

> 破坏性操作，默认 `no` 关闭。选择 `all` / `movie` / `tv` 后可能删除旧整理记录的源文件、媒体库目标文件，并发送文件删除事件移除历史下载的旧种子。

- **解决什么问题**：整季洗版下载前先清理旧版本整理记录和文件，避免新旧版本并存。
- **什么时候开或关闭**：确认媒体库路径、硬链接关系和旧版本清理策略都正确后，再把范围从 `no` 改为需要的媒体类型。对分集洗版、移动模式不熟悉或媒体库没有备份时保持 `no`。
- **典型联动**：仅整季洗版执行，分集洗版会跳过历史清理。范围可单独限定电影或剧集。
- **常见误用**：不要把它当成普通的记录清理选项。该能力会删除文件，风险高于只修改订阅状态。

<a id="cfg-verify_enabled"></a>
#### 自动纠错（`verify_enabled`）

- **解决什么问题**：订阅完成后 TMDB 可能继续新增集数。插件始终保存包含完成时总集数和订阅配置的完成快照；开启自动纠错后按周期复查，发现当前集数大于完成时集数时，会从下一集开始重建订阅。
- **什么时候开或关闭**：长篇动画、分割放送、绝对季和经常修改集数的剧集建议保持开启。只订阅电影或不希望自动重建订阅时可以关闭。
- **典型联动**：「自动纠错间隔（小时）」控制复查频率，「快照保留（天）」控制复查窗口。自动纠错不依赖完结守卫，订阅完成事件（`SubscribeComplete`）是唯一的完成快照入口。恢复时按 TMDB ID、季号和剧集组精确匹配。
- **常见误用**：完成快照过期后不会再复查。希望长期关注新增集数时，需要调大「快照保留（天）」。

## 使用步骤

1. 在插件市场安装「订阅助手（增强版）」。
2. 确认 MoviePilot 订阅使用 TMDB 数据源；如需消息提醒，先配置通知渠道。
3. 按需要开启「启用插件」，再选择「完结守卫模式」「自动暂停订阅」「自动纠错」等自动状态管理能力；完结守卫默认使用平衡模式，「下载超时自动删除」默认开启，启用前建议确认下载器可读且排除标签正确。
4. 按需要配置洗版，在界面中把「洗版类型」从「关闭」（`no`）改为目标范围。
5. 需要迁移已有洗版订阅时，先开启「回填已存在集」，再勾选一次「立即扫描存量并回填」。
6. 勾选「立即运行一次」保存，观察日志中的元数据、下载监控、洗版、完成后验证和通用巡检记录。
7. 运行一段时间后，再考虑把「清理整理记录范围」从 `no` 改为目标范围。

## 注意事项 / 已知风险

- **数据与文件风险**：插件会修改订阅状态、删除种子、写订阅历史和删除订阅。「清理整理记录范围」不是 `no` 时还可能删除源文件和媒体库文件。
- **仅支持 TMDB 数据源**：非 TMDB 订阅的集数、播出状态和完成判断可能不可靠。
- **不提供识别增强**：增强版不拦截同名真人与动漫、电影与剧集之间的互串资源。需要该能力时请使用旧版订阅助手。
- **洗版清理默认关闭**：不确认硬链接、媒体库路径和旧版本清理策略前，不要把「清理整理记录范围」改出 `no`。
- **标记暂停不会自动恢复**：无下载和用户名暂停需要用户手动启用或等待无下载策略处理。上映前和播出间隔暂停才会按元数据条件双向恢复。
- **一次性动作会自动复位**：「重置数据」「立即运行一次」「立即扫描存量并回填」执行后会自动取消勾选。
- **运行概况入口**：插件不提供详情页和仪表盘部件，可通过日志或 `/summary` API 查看运行概况。

## 故障排查

- 主日志：`MoviePilot/config/logs/moviepilot.log`
- 插件日志：`MoviePilot/config/logs/plugins/subscribeassistantenhanced/`
- 建议过滤词：`订阅助手（增强版）`、`subscribeassistantenhanced`、`注册定时任务`、`信号引擎`、`完成守卫`、`待定（P）`、`元数据巡检`、`资源选择`、`资源下载`、`下载器任务已建立`、`整理完成`、`下载监控`、`种子删除处理`、`近期删除资源清理`、`自动洗版`、`洗版巡检` 和 `完成后验证`
- 常见问题：
  - 没有看到定时任务 → 确认「启用插件」为开；再看日志里的“注册定时任务”摘要。
  - 普通订阅完成后没有自动洗版 → 确认「洗版类型」不是「关闭」（`no`），且媒体类型落在所选范围内；若选择剧集（分集下载），还需存在多条关联分集下载历史。
  - 洗版订阅没有继续推进 → 先看「洗版检查周期」是否有效，再确认订阅本身是洗版订阅、媒体识别成功、目标集是否已在库。若刚整理完成，也检查 `TransferComplete` / “整理完成”日志是否已经触发即时转全集。
  - 待定长期不释放 → 检查「待定超时释放（天）」和日志中的 total_episode 变更；数据仍在变化时会重置释放计时。低置信完结需要观察到期后再次命中同一信号才会完成。
  - 下载超时没有删种 → 确认下载器状态能读取、种子没有「排除标签」、进度窗口内增量是否超过阈值。
  - 手动删种没有触发处理 → 需要下载器可达且连续确认种子不存在；单次连接抖动不会立即当作手动删除。
  - Tracker 关键字不生效 → 确认「监听Tracker响应关键字」开启，并在「打开Tracker配置窗口」里逐行填写关键字或正则表达式。
  - 误删或误暂停 → 先关闭相关 Tab 的开关或调大阈值，再用「重置数据」清理插件侧任务记录；涉及文件删除的场景需从备份或媒体库管理工具恢复。

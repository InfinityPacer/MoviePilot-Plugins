写作约定：

- **去 AI 味**。下面这些写法是 AI 生成内容的典型痕迹，写作时主动规避；评审时遇到要打回：
  - **暴露"我是怎么知道的"**：写"源码顶部 alert 标注 / 源码 line N / `get_form` 返回 /
    `init_plugin` 默认值 / 表单 hint 里写的"等措辞。读源码是写作要求，不是用户需要看到的内容；
    直接给结论，例如写"实验性功能"，而不是"源码顶部 alert 明确标注为实验性功能"。
  - **示例里的免责声明**：括号备注"所有 value 均为表单实际枚举 / 已与源码核对 / 仅为示意"等。
    用户默认就期望你给的就是对的，写出来反而让人怀疑。
  - **commit-message 视角**：把"v2.12 起 / 自 v2.14 起新增 / 已修复 #1234"这类版本/issue
    引用塞进面向用户的描述段落。版本号属于"版本更新日志"小节，正文里直接讲当前行为。
  - **结构化过头**：连续段落都用"→"或 emoji 引导分支、句句"建议/强烈建议/务必"叠加、
    每个要点都按"原因 / 结论 / 注意"三段式排版。骨架统一是为了便于扫读，不是为了机械整齐——
    保留五段骨架，但每一段内部允许写成自然句。
  - **没有信息的导语**：表格 / 列表上方一句"下表合并列出关键配置项"或"以下是 N 条要点"这种

  - **以 AI 视角解释自己的输出**：例如"为了帮助用户理解，本节将…"。用户不需要这种铺垫。

理解配置含义的强制要求（关键）：
- 写"配置说明"和"深入说明"之前，**必须**先读源码再推理，不允许只看表单 `hint` 或字段名猜含义。
  表单 hint 是写给已经懂的人看的速记，直接翻译 hint 几乎一定会复现"用户看不懂"的问题。
- 至少要看的源码范围：
  1. 插件本目录的 `__init__.py`（重点：`init_plugin` 里 `config.get(...)` 的**兜底默认值**与类型转换，
     `get_form` 里该字段所在的 Tab / 联动控件 / 可选枚举 / 隐藏条件，
     `get_form` 返回的第二个元素那个 form-default 字典——这才是用户首次打开表单看到的初值，
     该字段在业务方法里的真实使用路径——只有用法才说明配置真正在做什么）；
  2. 插件本目录其它模块（如 `recognition_guard.py` 这类辅助逻辑文件，往往承载真正的判定规则）；
  3. 主程序中被该配置影响到的链路（订阅 / 下载 / 整理 / 通知等模块），用来判断配置改动后
     在主流程的哪个阶段生效、与主程序默认行为是覆盖还是叠加；
  4. `package.v2.json` 中该插件的 `history`，结合版本日志判断该字段是新增、改动还是兼容遗留。
- 默认值要以 **form-default 为准**（即 `get_form` 返回的 `(content, defaults)` 元组里那个
  `defaults` 字典）。不要拿 `init_plugin` 里 `config.get("k", X)` 的兜底 X 直接当默认值——
  兜底只在配置项缺失时生效，常与 form-default 不一致（例如表单 default=24，但兜底 default=0
  用来表达"未配置时按禁用处理"）。两者不一致时，"默认值"列写 form-default，"备注"列补一句
  兜底语义（如"配置为 0 表示禁用"）。
- 枚举字段必须读 `get_form` 里 `'items': [...]` 的真实枚举值列表（中文 title + 真实 value），
  不要凭字段名或 hint 中的"观察 / 拦截"自己拼三档；同时去业务代码里搜索 `{...枚举集合...}`
  来确认合法值集合——常有比表单更多的历史兼容值。**严禁在示例里写不存在的枚举 value。**
- 单位（小时 / 分钟 / 天 / 百分比）要在源码里找乘数确认（如 `* 3600` 是小时、`timedelta(days=...)`
  是天），不要凭字段名后缀（`_days` 不一定就是天）。
- 输出时把"读源码推理出来"的结论写进表格 + 深入说明；不要保留"按需启用 / 视情况而定 / 详见 hint"
  这类空话，否则视同未理解。
- 如果读完源码仍无法判断某字段的业务含义（例如纯实验性开关），在该字段的"备注"列显式写明
